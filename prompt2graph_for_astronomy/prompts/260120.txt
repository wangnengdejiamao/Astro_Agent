Your task is to extract triples and attributes that are **semantically faithful**, **schema-aligned**, and **minimally redundant**, with full traceability to the source text.

### CORE PRINCIPLES (STRICTLY ENFORCED)

#### 1. ENTITY CANONICALIZATION BASED ON SCHEMA TYPE

The final name of every entity must reflect its **semantic role** AND align with its assigned **schema type**:

##### A. Component-Level Entities (`ChemicalCompound`, `Salt`, `Solvent`, `Additive`)
- Extract **only the canonical chemical substance name**, stripping ALL contextual modifiers:
  - ❌ Remove: concentrations ("0.2 M", "5 wt%"), ratios ("3:7"), states ("anhydrous", "hexahydrate" → keep as attribute), procedures ("added", "dissolved"), qualifiers ("excess", "trace").
  - ✅ Keep: standardized chemical name or formula.
- Normalize formulas to ASCII (PubChem-style):
  - `$In(NO_{3})_{3}·6H_{2}O$` → `In(NO3)3·6H2O`
  - `LiPF₆` → `LiPF6`
  - `EC` → retain if standard abbreviation; otherwise use `ethylene carbonate`

  Examples:
  - Source: "0.2M In(NO₃)₃·6H₂O was used."  
    → Entity: `"In(NO3)3"` (type: `Salt`)  
    → Attributes: `concentration: 0.2 M`, `hydration_state: hexahydrate`
  - Source: "EC/DMC (3:7 v/v)"  
    → Entities: `"ethylene carbonate"` (type: `Solvent`), `"dimethyl carbonate"` (type: `Solvent`)  
    → Attribute on electrolyte: `ratio: EC:DMC = 3:7`

##### B. System-Level Entities (`Electrolyte`, `Anode`, `Cathode`)
- Preserve **essential compositional or structural descriptors** that define the system:
  - ✅ Keep: "electrolyte with FEC", "NMC811 cathode", "Si anode coated with carbon"
  - ❌ Remove: performance claims ("showing high stability"), measurement results ("with 100 ppm water")

##### C. `Property` and `PerformanceMetric`
- Use **standardized, value-free names**:
  - ✅ `"water content"`, `"ionic conductivity"`, `"Coulombic efficiency"`
  - ❌ `"conductivity of 8.5 mS/cm"`, `"efficiency >99%"`

##### D. Consistency
- Use the **same canonical name** for an entity across triples, attributes, and `entity_types`.

##### E. Abbreviation and Synonym Handling (CRITICAL FOR ENTITY DEDUPLICATION)

When the same chemical substance appears with **different names, abbreviations, or formatting variations**, you MUST:

1. **Identify all variants** of the same entity:
   - **Abbreviations**: `"EC"` ↔ `"ethylene carbonate"`, `"DMC"` ↔ `"dimethyl carbonate"`, `"FEC"` ↔ `"fluoroethylene carbonate"`
   - **Synonyms**: `"LiPF6"` ↔ `"lithium hexafluorophosphate"`, `"NMC811"` ↔ `"LiNi0.8Mn0.1Co0.1O2"`
   - **Case variations**: `"LiPF6"` ↔ `"lipf6"` ↔ `"LiPF₆"` (normalize to standard case/formula)
   - **Formatting differences**: `"Cu(NO3)2"` ↔ `"Cu(NO₃)₂"` ↔ `"copper nitrate"` (normalize to ASCII formula)

2. **Choose ONE canonical name** as the primary entity name:
   - Prefer: **standardized chemical formula** (e.g., `"LiPF6"`, `"Cu(NO3)2"`) or **IUPAC/common name** (e.g., `"ethylene carbonate"`)
   - Use the **most complete/full form** when multiple variants appear in the text
   - For abbreviations, prefer the **full name** if both appear, otherwise use the **most commonly used abbreviation**

3. **Extract abbreviation relationships** as attributes:
   - For each variant that differs from the canonical name, add an `abbreviation` attribute
   - Format: `["abbreviation: <variant_name>", source, evidence]`
   - Include ALL variants (abbreviations, synonyms, case variations) as separate abbreviation entries

4. **Examples**:

   Example 1: Abbreviation and full name
   - Source: "EC (ethylene carbonate) and DMC were mixed."
   - Canonical entity: `"ethylene carbonate"` (type: `Solvent`)
   - Attributes: `["abbreviation: EC", "EC (ethylene carbonate)", "Standard abbreviation for ethylene carbonate"]`
   - Also extract: `"dimethyl carbonate"` (type: `Solvent`) with `["abbreviation: DMC", ...]`

   Example 2: Formula and name
   - Source: "LiPF6 (lithium hexafluorophosphate) was added."
   - Canonical entity: `"LiPF6"` (type: `Salt`) [prefer formula]
   - Attributes: `["abbreviation: lithium hexafluorophosphate", "LiPF6 (lithium hexafluorophosphate)", "Full chemical name for LiPF6"]`

   Example 3: Case variations
   - Source: "The electrolyte contained LiPF6 and lipf6 was also tested."
   - Canonical entity: `"LiPF6"` (type: `Salt`) [standardize to proper case]
   - Attributes: `["abbreviation: lipf6", "lipf6 was also tested", "Case variation normalized to LiPF6"]`

   Example 4: Multiple variants
   - Source: "FEC (fluoroethylene carbonate, 2% v/v) and fluoroethylene carbonate were used."
   - Canonical entity: `"fluoroethylene carbonate"` (type: `Additive`) [prefer full name]
   - Attributes: 
     - `["abbreviation: FEC", "FEC (fluoroethylene carbonate)", "Standard abbreviation"]`
     - `["concentration: 2% v/v", "FEC (fluoroethylene carbonate, 2% v/v)", "Concentration attribute"]`

5. **Critical rules**:
   - ✅ **ALWAYS** extract abbreviation relationships when variants appear in the same or adjacent sentences
   - ✅ **ALWAYS** use the same canonical name in `entity_types` and all triples
   - ✅ **ALWAYS** include abbreviation attributes for ALL variants found in the text
   - ❌ **NEVER** create separate entities for abbreviations/variants — they must share the same canonical name
   - ❌ **NEVER** mix variants in entity names (e.g., don't use "EC/ethylene carbonate" as one entity)

#### 2. TRIPLE DIRECTION MUST RESPECT GRAMMATICAL LOGIC

- In passive voice (**"X was [verb]ed by Y"**, **"X measured using Y"**):
  - X = logical object (e.g., property, sample)
  - Y = agent/instrument (e.g., method)
  - Triple: `[X, relation, Y]`

  ✅ Correct:  
  `["water content", "is_measured_by", "Karl Fischer titration"]`  
  ❌ Wrong:  
  `["Karl Fischer titration", "is_measured_by", "water content"]`

- For active voice: follow subject–verb–object order.

---

#### 3. SOURCE = CONTIGUOUS, VERBATIM CONTEXT

- The `source` must be a **contiguous excerpt** from the input text (1–2 sentences or even the entire paragraph) that **explicitly mentions both the subject and object**.
- It should provide **sufficient context** for a domain expert to verify the relationship without external knowledge.
- Do NOT truncate or paraphrase.

---

#### 4. EVIDENCE = EXPLICIT REASONING

In `evidence`, explain:
- The **grammatical structure** (active/passive, relative clause, etc.)
- Why the **entity name was canonicalized** (e.g., “removed concentration per schema rule for Salt”)
- Why the **triple direction is correct**
- Why a **simpler or more complex phrase would be wrong**

Example:
> Source: "The water concentration (~100 ppm) was measured by Karl Fischer titration."
> Evidence: "Passive construction: 'water concentration' is the logical object of 'measured', and 'Karl Fischer titration' is the method. Per schema, 'water concentration' is a Property, and the method is a TestMethod. Concentration value (~100 ppm) is extracted as an attribute, not part of the entity name."

---

#### 5. OUTPUT FORMAT

Return ONLY a JSON object with:
- `"triples"`: list of `[subject, relation, object, source, evidence]`
- `"attributes"`: dict `{entity: [["key: value", source, evidence], ...]}`
  - **MUST include `abbreviation` attributes** for all variants (abbreviations, synonyms, case variations) of the same entity
  - Format: `["abbreviation: <variant_name>", source, evidence]`
- `"entity_types"`: dict `{entity_name: schema_type}`

All entity names must be **canonicalized per Section 1**, and **abbreviation relationships must be extracted per Section 1.E**.
