#### 2. ATTRIBUTE EXTRACTION PRINCIPLES (CRITICAL)

##### 2.1 ONE ENTITY, ONE ATTRIBUTE INSTANCE

**CRITICAL**: For each entity, each type of attribute should appear ONLY ONCE.

- ❌ WRONG: Multiple separate attributes for the same measurement:
  ```json
  "ionic conductivity": [
    {"key": "value", "value": "10.8 mS cm⁻¹"},
    {"key": "description", "value": "room-temperature"},
    {"key": "condition", "value": "25°C"}
  ]
  ```

- ✅ CORRECT: Merge into ONE comprehensive attribute:
  ```json
  "ionic conductivity": [
    {
      "key": "value",
      "value": "10.8 mS cm⁻¹ at room-temperature (25°C)"
    }
  ]
  ```

**Rules**:
- Combine **numeric value + units + conditions + descriptions** into a single `value` attribute
- Use natural language to connect components: "at", "under", "measured by", "vs."
- Preserve all information but in ONE entry

##### 2.2 COMPREHENSIVE VALUE FORMAT

When extracting measurement values, include ALL contextual information:

| Component | Example | Include? |
|-----------|---------|----------|
| Numeric value | "10.8" | ✅ Always |
| Units | "mS cm⁻¹" | ✅ Always |
| Conditions | "room-temperature", "at 25°C" | ✅ Merge |
| Comparison reference | "vs. Li/Li⁺" | ✅ Merge |
| Method | "measured by ICP-OES" | ✅ Merge (if space allows) |

**Examples of CORRECT merged values**:
- `"10.8 mS cm⁻¹ at room-temperature"`
- `"172 °C (thermal decomposition onset)"`
- `"4.6 V vs. Li/Li⁺ (electrochemical window)"`
- `"99.95% purity confirmed by ICP-OES"`

#### 3. ATTRIBUTE TYPES

Extract attributes based on the schema definition:

- **value**: Complete measurement with all context (e.g., "10.8 mS cm⁻¹ at room-temperature", "172 °C decomposition onset")
  - ✅ **CRITICAL**: ALWAYS merge value + units + conditions + descriptions into ONE string
  - ❌ Do NOT extract: separate "value", "units", "description", "condition" entries
  - ✅ DO extract: single comprehensive value string

- **concentration**: "0.2 M", "5 wt%", "2% v/v"
  - For concentration, context is usually not needed, keep as-is

- **abbreviation**: abbreviations/variants (use abbreviation_mappings from Stage 1)
  - Keep as separate attribute (different purpose)

- **hydration_state**: "hexahydrate", "trihydrate", "anhydrous"
  - Keep as separate attribute (intrinsic property)

- **formula**: chemical formula
  - Keep as separate attribute (intrinsic property)

#### 5. SOURCE AND EVIDENCE

- **source**:
  - Must be a **contiguous, verbatim excerpt** from the input text that clearly contains the attribute value.
  - **Prefer full sentences** (1–2 sentences), starting from the beginning of the sentence and ending at the sentence-final punctuation (e.g., `.`, `!`, `?`, or the equivalent in other languages).
  - **Do NOT use only a truncated clause or verb phrase**. For example:  
    - ❌ Do NOT use: `"shows a specific capacity of 167 mAhg^{-1}"` (missing subject and full sentence)  
    - ✅ DO use: `"The cell shows a specific capacity of 167 mAhg^{-1} at 0.1 C."` or the entire sentence/paragraph containing this statement.

- **evidence**:
  - Explain **why** this attribute is extracted (e.g., `"Concentration value extracted as attribute per Salt canonicalization rule"`).
  - Explain **how** the chosen source supports this attribute.
  - For merged values, explain what components were combined (e.g., `"Value 10.8, units mS cm⁻¹, and condition room-temperature merged into single measurement description"`).

---

#### 6. DEDUPLICATION (CRITICAL)

**Before extracting an attribute, check if an identical or highly similar attribute already exists for this entity.**

- If the SAME entity with the SAME key and SIMILAR value exists: **MERGE or SKIP**
- If the value differs significantly (e.g., two different concentrations): **KEEP BOTH**

**Examples**:
- ❌ AVOID: Two entries "10.8 mS cm⁻¹" and "10.8 mS cm⁻¹ at room-temperature" → MERGE into one
- ✅ ALLOW: "0.1 M" and "0.5 M" for the same entity (different concentrations)
