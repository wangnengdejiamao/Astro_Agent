 CORE TASK: RELATION EXTRACTION

 

This extraction is strictly sample-centered.

 

The ExperimentalSample is the primary factual unit in experimental papers.

 

In Stage 2, your goal is to connect the Stage 1 entities using schema-defined relations only.

 

Do NOT create attributes in this stage.

Do NOT convert scalar values into relation objects.

Do NOT invent new entity names unless absolutely necessary because Stage 1 missed an entity that is explicitly required by the text.

 

 

 \1. USE CANONICAL ENTITY NAMES FROM STAGE 1

 

CRITICAL: You MUST use ONLY the canonical entity names identified in Stage 1.

 

\- Do NOT use variant names.

\- Do NOT use abbreviation-only names if Stage 1 already provided the canonical name.

\- Do NOT rename entities.

\- Do NOT replace canonical entity names with generic referents such as:

  \- "the sample"

  \- "this material"

  \- "the absorber"

  \- "the alloy"

  \- "the support"

 

If a generic referent clearly points to an entity already identified in Stage 1, use that Stage 1 canonical entity name.

 

If the text clearly expresses a relation involving an entity that Stage 1 should have recognized but omitted, you may still extract the triple using the most appropriate canonical form under the same naming rules as Stage 1. However, prefer Stage 1 canonical names whenever available.

 

 

 \2. USE ONLY RELATIONS DEFINED IN THE SCHEMA

 

You MUST use only relation types that are present in the schema.

 

For the sample-centered EM absorber schema, the key relations include:

\- has_component

\- supported_on

\- derived_from_system

\- prepared_by

\- processed_under

\- has_structure_concept

\- characterized_by

\- supports_finding

\- associated_with_mechanism

\- limited_by

\- compared_with

 

Do NOT invent free-form relation names outside the schema.

 

Do NOT replace schema relations with looser paraphrases.

 

 

 \3. RELATION DIRECTION MUST FOLLOW SCIENTIFIC SEMANTICS

 

Relation direction must follow scientific semantics and local grammatical logic.

 

 A. Backbone direction

Use the following directions:

 

\- MaterialSystem -> has_component -> MaterialComponent

\- MaterialSystem -> supported_on -> CarbonSupport

\- ExperimentalSample -> derived_from_system -> MaterialSystem

\- ExperimentalSample -> prepared_by -> SynthesisMethod

\- ExperimentalSample -> processed_under -> ProcessingCondition

\- ExperimentalSample -> has_structure_concept -> StructureConcept

\- ExperimentalSample -> characterized_by -> CharacterizationMethod

\- CharacterizationMethod -> supports_finding -> StructureConcept / LossMechanism / Limitation

\- ExperimentalSample -> associated_with_mechanism -> LossMechanism

\- ExperimentalSample -> limited_by -> Limitation

\- ExperimentalSample -> compared_with -> ExperimentalSample

 

Never reverse these relations unless the schema explicitly defines the reverse form.

 

Examples:

\- Correct: Sample_FeCu_GDY_T1000 -> derived_from_system -> System_FeCu_GDY

\- Wrong: System_FeCu_GDY -> derived_from_system -> Sample_FeCu_GDY_T1000

 

\- Correct: XRD -> supports_finding -> StructureConcept_Cu_Peak_Weakening

\- Wrong: StructureConcept_Cu_Peak_Weakening -> supports_finding -> XRD

 

 B. Active and passive constructions

When determining the correct triple direction, follow the logical subject-object structure of the text.

 

\- Active voice: follow subject-verb-object order

\- Passive or evidence statements: preserve the scientific role of the entities rather than surface word order

 

For example:

\- "The sample was characterized by XRD"  

  -> Sample -> characterized_by -> XRD

\- "XRD revealed weakening of Cu peaks"  

  -> XRD -> supports_finding -> StructureConcept_Cu_Peak_Weakening

 

 

 \4. SAMPLE-CENTERED RELATION EXTRACTION PRINCIPLES

 

 A. Always connect sample-level findings back to ExperimentalSample

If a structure, mechanism, or limitation is explicitly attributed to a specific tested sample, connect that finding to the corresponding ExperimentalSample.

 

Examples:

\- Sample_FeCu_GDY_T1000 -> has_structure_concept -> StructureConcept_Enhanced_Alloying

\- Sample_FeCu_GDY_T1100 -> limited_by -> Sintering

\- Sample_FeCu_GDY_T800 -> associated_with_mechanism -> Dielectric_Relaxation

 

Do NOT leave key findings floating without a sample connection if the text clearly assigns them to a specific sample.

 

 B. Preserve MaterialSystem / ExperimentalSample distinction

Use MaterialSystem only for platform-level relations such as composition/support structure.

 

Use ExperimentalSample when the relation depends on explicit experimental conditions or measured sample-level findings.

 

Examples:

\- System_FeCu_GDY -> has_component -> FeCu

\- System_FeCu_GDY -> supported_on -> GDY

\- Sample_FeCu_GDY_T1000 -> has_structure_concept -> StructureConcept_Enhanced_Alloying

 

Do NOT attach condition-specific findings directly to MaterialSystem unless the text is genuinely describing the entire platform in a non-sample-specific way.

 

 

 \5. CHARACTERIZATION-SUPPORTED RELATION RULES

 

Characterization information must be extracted as an evidence chain when explicitly supported by the text.

 

If a characterization method is explicitly linked to a finding, preserve BOTH:

\- the sample-to-method relation

\- the method-to-finding relation

 

Preferred patterns:

\- ExperimentalSample -> characterized_by -> CharacterizationMethod

\- CharacterizationMethod -> supports_finding -> StructureConcept / LossMechanism / Limitation

\- ExperimentalSample -> has_structure_concept / associated_with_mechanism / limited_by -> supported finding

 

 Specific guidance

 

 XRD

If XRD supports conclusions such as:

\- phase dominance

\- Cu peak weakening/disappearance

\- alloying

\- interdiffusion

\- phase evolution

 

then extract:

\- Sample -> characterized_by -> XRD

\- XRD -> supports_finding -> corresponding StructureConcept

\- and, when explicitly justified, Sample -> has_structure_concept -> corresponding StructureConcept

 

 ICP or elemental analysis

If ICP supports conclusions such as:

\- non-equiatomic composition

\- composition deviation

\- trace-element retention

 

then extract:

\- Sample -> characterized_by -> ICP

\- ICP -> supports_finding -> corresponding StructureConcept

\- and, when explicitly justified, Sample -> has_structure_concept -> corresponding StructureConcept

 

 Cole-Cole analysis

If Cole-Cole analysis supports dielectric-relaxation interpretation, then extract:

\- Sample -> characterized_by -> Cole_Cole_Analysis

\- Cole_Cole_Analysis -> supports_finding -> Dielectric_Relaxation

\- and, when explicitly justified, Sample -> associated_with_mechanism -> Dielectric_Relaxation

 

 Electron microscopy

If electron microscopy supports conclusions such as:

\- particle aggregation

\- dense morphology

\- poor dispersion

\- sintering

 

then extract:

\- Sample -> characterized_by -> Electron_Microscopy

\- Electron_Microscopy -> supports_finding -> corresponding StructureConcept or Limitation

\- and, when explicitly justified, Sample -> has_structure_concept / limited_by -> corresponding finding

 

Do NOT use supports_finding for raw scalar values such as particle size numbers.

Those values belong to Stage 3 attributes.

 

 

 \6. DO NOT COLLAPSE MULTI-STEP SCIENTIFIC LOGIC

 

When the text contains an explicit multi-step scientific chain, preserve the intermediate node(s) when they exist in Stage 1 and are explicitly supported by the text.

 

For example, if the text states in effect:

\- a sample shows enhanced alloying

\- enhanced alloying is associated with magnetic loss

\- excessive temperature causes sintering and poor dispersion

 

then do NOT collapse everything into one vague relation.

 

Prefer explicit chains such as:

\- Sample_FeCu_GDY_T1000 -> has_structure_concept -> StructureConcept_Enhanced_Alloying

\- Sample_FeCu_GDY_T1000 -> associated_with_mechanism -> Magnetic_Loss

\- Sample_FeCu_GDY_T1100 -> limited_by -> Sintering

\- Sample_FeCu_GDY_T1100 -> limited_by -> Poor_Dispersion

 

If a characterization method is the explicit support for a finding, keep that support edge too:

\- XRD -> supports_finding -> StructureConcept_Enhanced_Alloying

\- Electron_Microscopy -> supports_finding -> Sintering

 

Never reduce an explicit evidence chain to only one broad summary triple when the intermediate finding is explicitly present in the text.

 

 

 \7. COMPARED_WITH RELATIONS

 

Use compared_with only when the text explicitly compares two samples, systems, methods, or findings.

 

For this schema, compared_with will most often be used between ExperimentalSample entities.

 

Examples:

\- Sample_FeCu_GDY_T600 -> compared_with -> Sample_FeCu_GDY_T1000

\- Sample_FeCu_GDY_L30wt -> compared_with -> Sample_FeCu_GDY_L50wt

 

Do NOT create compared_with merely because two entities appear in the same sentence.

The comparison must be explicit or strongly structured by the text.

 

 

 \8. RELATION EXTRACTION COMPLETENESS

 

Extraction is incomplete if:

\- a clearly stated backbone relation is omitted

\- a sample is explicitly characterized by a method but Sample -> characterized_by -> Method is omitted

\- a characterization-supported finding is explicit but Method -> supports_finding -> Finding is omitted

\- a finding is explicitly assigned to a sample but the Sample -> finding relation is omitted

 

This is especially important for:

\- XRD

\- ICP

\- Cole-Cole analysis

\- Electron microscopy

 

Do not stop at extracting only the method name if the text explicitly gives the supported scientific conclusion.

 

 

 \9. SOURCE REQUIREMENTS

 

For each triple:

 

\- `source` must be a contiguous, verbatim excerpt from the input text.

\- It should explicitly mention both the subject and object, or provide enough local context for a domain expert to verify the relation.

\- Prefer one or two full sentences, or a compact paragraph span when needed.

\- Do NOT paraphrase.

\- Do NOT use fragmented non-interpretable snippets if a full sentence is available.

 

 

 \10. EVIDENCE REQUIREMENTS

 

For each triple, `evidence` must briefly explain:

 

\- why the relation is supported by the text

\- why the subject and object canonical names from Stage 1 are appropriate

\- why the triple direction is correct

\- if relevant, why a more collapsed or reversed relation would be wrong

\- if relevant, why an intermediate evidence-chain triple is preserved rather than omitted

 

Keep evidence concise but explicit.

Return ONLY a JSON object with:

  \- `"relation"`: schema relation

  \- `"object"`: canonical entity name from Stage 1

  \- `"source"`: verbatim text excerpt

  \- `"evidence"`: brief reasoning for the relation and direction
