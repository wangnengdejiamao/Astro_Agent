==================================================
1. EXTRACTION ORDER (MANDATORY)
==================================================

Always follow this order:

Step 1. Identify SourceDocument boundaries.
Step 2. Identify one or more true MaterialSystem nodes under each SourceDocument.
Step 3. Identify all ExperimentalSample mentions belonging to those systems.
Step 4. Resolve all sample aliases, shorthand names, descriptive mentions, and pronouns into canonical sample identities.
Step 5. Attach explicit sample-specific scalar facts to the merged ExperimentalSample nodes.
Step 6. Attach shared family-level non-scalar findings to the MaterialSystem.
Step 7. Attach characterization / finding / structure / property / mechanism / limitation nodes around the sample ring.
Step 8. Reuse globally canonical component/support/concept/property/mechanism nodes across papers to increase connectivity.

Never output sample facts before completing identity resolution.

==================================================
2. MULTI-DOCUMENT ANCHORING RULE
==================================================

The final graph is global across many papers.
Do NOT try to keep papers isolated.
But every ExperimentalSample must still be anchored to:
- exactly one SourceDocument
- exactly one primary MaterialSystem

If the chunk contains multiple papers, titles, labels such as “Paper 1”, “Paper 2”, or clearly distinct study systems:
- create separate SourceDocument nodes
- create the correct MaterialSystem nodes under the correct SourceDocument
- assign each sample to the correct SourceDocument and MaterialSystem
- do NOT mix samples from different papers into the same MaterialSystem

Use SourceDocument as an anchor, not as a barrier.
Cross-paper connectivity should happen through shared canonical concept nodes, not by merging unrelated samples.

Preferred SourceDocument names:
- Doc_1
- Doc_2
- Doc_3

==================================================
3. ONE ENTITY = ONE PRIMARY TYPE RULE
==================================================

Each entity must have exactly one primary schema type.

Do NOT create:
- one node with multiple schema types
- one node whose name is a list/array of names
- one node serving simultaneously as MaterialSystem and MaterialComponent
- one node serving simultaneously as ExperimentalSample and MaterialComponent

Examples:
- pristine graphene used as an experimentally measured comparator -> ExperimentalSample
- graphene used as the support/substrate within another sample -> CarbonSupport
- Co SAs inside Co SA/graphene -> MaterialComponent
- SA-graphene family/platform -> MaterialSystem
- Model 1 / Model 2 predicted recipes -> DesignOutput

If the same surface string can refer to different roles in different contexts, choose the correct type for the local context rather than merging unlike types.

==================================================
4. MATERIAL SYSTEM RULE
==================================================

A MaterialSystem is the central experimental platform or sample family within one SourceDocument.

Use MaterialSystem for:
- FeCu/GDY family
- FeCoNiCuZn/GDY family
- Fe3C microflake family
- Co SA/graphene family
- HCNF/HEA family

A paper may contain more than one MaterialSystem only if the paper truly studies multiple distinct central systems.

Do NOT create MaterialSystem nodes for:
- background-only literature examples
- generic review phrases
- author names
- vague words such as “materials”, “composites”, “supports”
unless a more specific system name truly cannot be formed and the phrase is central to the paper’s own experimental platform.

Each MaterialSystem should connect to:
- one SourceDocument
- at least one ExperimentalSample
- and at least one of:
  - MaterialComponent
  - CarbonSupport
  - StructureConcept
  - LossMechanism
  - EMProperty

ExperimentalSample nodes must surround and derive from MaterialSystem nodes.

Every ExperimentalSample must:

==================================================
6. MANDATORY SAMPLE MERGING / COREFERENCE RULE
==================================================

This is the highest-priority identity rule.

If multiple mentions refer to the same experimental sample, they MUST be merged into one canonical ExperimentalSample node.

Merge when they refer to the same sample through:
- full name vs short name
- sample code vs descriptive phrase
- “the sample”
- “this sample”
- “it”
- “the 800°C sample”
- “the 50 wt% loading sample”
- “the optimal sample”
- “the Cu-only/GDY sample”
- “the counterpart”
- repeated mentions across neighboring sentences
- formatting variants
- OCR variants
- family label + unique condition label

Examples:
- “sample annealed at 800°C for 4 h”
- “the 800°C sample”
- “this sample”
-> one canonical ExperimentalSample

- “50 wt% loading sample”
- “the optimal loading sample”
-> one canonical ExperimentalSample, e.g. Sample_FeCu_GDY_L50

- “Cu-only/GDY sample”
- “Cu-only counterpart”
-> one canonical ExperimentalSample if context indicates same object

Use:
- sample_name = one canonical name only
- sample_aliases = all alternate surface forms

Never create duplicate samples that differ only by:
- wording
- abbreviation
- formatting

11. FIELD LEGALITY RULE
==================================================

Use scalar fields conservatively and legally.

Allowed:
- RL_min_value -> one numeric dB value only
- RL_min_frequency -> one numeric GHz point only
- EAB_value -> one numeric GHz width only
- EAB_frequency_range -> one explicit GHz interval only
- matching_thickness -> one absorber thickness only
- particle_size -> one explicit size value only
- filler_loading -> wt% loading only
- metal_molar_ratio -> true composition ratio only
- precursor_loading_ratio -> true ratio only
- precursor_amount_note -> raw amounts such as “1.5 mmol CuCl, 500 mg TEB”

Forbidden:
- descriptive phrases in numeric fields
- absorption rate stored in RL_min_value
- “below 70%” in RL_min_value when it is actually absorption_rate
- “98%” in RL_min_value if it is an absorption peak
- “much narrower” in EAB_value
- “significantly smaller” in particle_size
- “specified” in filler_loading
- “500 mg TEB” in filler_loading
- “1.5 mmol CuCl” in metal_molar_ratio if it is not a ratio
- “outer diameter / inner diameter” in matching_thickness
- peak-position-only text in epsilon_real / epsilon_imag / mu_real / mu_imag unless clearly a value field

If a statement is descriptive rather than a legal scalar value, place it in:
- performance_note
- property_note
- remarks

Examples:
- “much narrower EAB” -> performance_note
- “significantly smaller particle size” -> property_note
- “complete absorption of X-band” -> performance_note
- “500 mg TEB” -> precursor_amount_note

==================================================

To improve cross-paper connectivity, globally reuse the same canonical node names for the same reusable concept when the identity is truly the same.

This applies especially to:
- MaterialComponent (e.g. Fe, Co, Ni, Cu, Mn, Fe3C, CuCl, TEB)
- CarbonSupport (e.g. graphene, graphdiyne, HCNF)
- StructureConcept (e.g. honeycomb-like porous structure, orthorhombic crystal structure, high aspect ratio, lattice distortion)
- EMProperty (e.g. reflection loss, effective absorption bandwidth, complex permittivity, complex permeability, electrical conductivity)
- LossMechanism (e.g. dielectric relaxation, interfacial polarization, magnetic loss, polarization loss, conduction loss)

Do NOT create minor wording variants as separate nodes when they clearly refer to the same reusable concept.
Prefer one canonical name.

Examples:
- “honeycomblike porous structure” and “honeycomb-like porous structure” -> one node
- “dielectric-loss-dominated mechanism” and “dielectric loss” -> canonicalize carefully if same mechanism intent
- “graphene” and “graphene oxide” are NOT the same unless the paper explicitly treats them as the same object
