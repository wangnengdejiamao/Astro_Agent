1. EXTRACTION ORDER (MANDATORY)
==================================================

Always follow this order:

Step 1. Identify SourceDocument boundaries.
Step 2. Identify one or more true MaterialSystem nodes under each SourceDocument.
Step 3. Identify all ExperimentalSample mentions belonging to those systems.
Step 4. Resolve all sample aliases, shorthand names, descriptive mentions, and pronouns into canonical sample identities.
Step 5. Attach explicit sample-specific scalar facts to the merged ExperimentalSample nodes.
Step 6. Attach shared family-level non-scalar findings to the MaterialSystem.
Step 7. Attach characterization / finding / structure / property / mechanism / limitation nodes around the sample ring.
Step 8. Reuse globally canonical component/support/concept/property/mechanism nodes across papers.

Never output sample facts before identity resolution is complete.

4. ONE ENTITY = ONE PRIMARY TYPE RULE
==================================================

Each entity must have exactly one primary schema type.

Never create:

- one node whose name is a list/array

If the same surface text can play different roles in different contexts, choose the type from the local context rather than merging unlike types.

9. OWNER DISCIPLINE MATRIX
==================================================

Use this ownership matrix strictly.

A. MaterialSystem owns:
- shared family-level non-scalar facts
- shared morphology
- shared structure
- shared mechanism tendency
- family identity
- shared performance summary only when not sample-specific

B. ExperimentalSample owns:
- all explicit measured scalar facts
- all sample-specific conditions

- all sample-specific characterization findings
- all sample-specific performance and mechanisms when directly assigned

C. MaterialComponent owns:
- chemical species
- alloy elements
- embedded particles/components
- precursor chemicals
- component-level composition facts

D. CarbonSupport owns:
- graphene / graphdiyne / HCNF / carbon scaffold identity when functioning as support/substrate

Hard examples:
- pristine graphene with measured absorption -> ExperimentalSample

- Co SAs inside Co SA/graphene -> MaterialComponent
- HEA nanoparticles inside HCNF/HEA -> MaterialComponent
- size/performance values tied to experimental condition -> ExperimentalSample, not MaterialComponent
- TEB -> MaterialComponent (precursor), never CarbonSupport
- paraffin matrix in loading tests -> MaterialComponent, but loading value belongs to ExperimentalSample.filler_loading

11. FIELD LEGALITY RULE
==================================================

Use scalar fields conservatively and legally.

Allowed:
- RL_min_value -> one numeric dB value only
- RL_min_frequency -> one numeric GHz point only
- EAB_value -> one numeric GHz width only
- EAB_frequency_range -> one explicit GHz interval only
- matching_thickness -> one absorber thickness only
- particle_size -> one explicit size only
- filler_loading -> wt% loading only
- metal_molar_ratio -> true ratio only
- precursor_loading_ratio -> true ratio only
- precursor_amount_note -> raw amounts such as “1.5 mmol CuCl, 500 mg TEB”

Forbidden:
- descriptive phrases in numeric fields
- absorption rate stored in RL_min_value
- “below 70%” in RL_min_value when it is actually absorption_rate
- “98%” in RL_min_value when it is actually absorption_rate
- “much narrower” in EAB_value
- “significantly smaller” in particle_size
- “specified” in filler_loading
- “500 mg TEB” in filler_loading
- “1.5 mmol CuCl” in metal_molar_ratio if it is not a ratio
- “outer diameter / inner diameter” in matching_thickness
- peak-position-only descriptions in epsilon_real / epsilon_imag / mu_real / mu_imag unless clearly a value

If a statement is descriptive rather than a legal scalar value, place it in:
- performance_note
- property_note
- remarks

Hard examples:
- “much narrower EAB” -> performance_note
- “significantly smaller particle size” -> property_note
- “500 mg TEB” -> precursor_amount_note
- “complete absorption of X-band” -> performance_note
- “0.9 at 11.56 GHz” if describing magnetic-loss ability -> NOT RL_min_value; keep as EMProperty or property_note unless schema field truly matches
